전반적 평가: 주의 필요 (Medium Risk)

chatbot.js는 XSS 방어를 위한 escapeHtml 함수, Shadow DOM 격리, textContent 기반 사용자 입력 렌더링 등 기본적인 보안 관행을 잘 따르고 있습니다. 그러나 LLM 응답을 innerHTML로 직접 렌더링하는 경로와 외부 URL에 대한 검증 부재, 입력 길이 제한 미적용 등 실제 공격으로 이어질 수 있는 취약점이 발견되었습니다. High 1건을 포함한 총 7건의 보안 이슈를 조치하면 프로덕션 배포에 적합한 수준이 됩니다.

🔴 발견된 취약점 (7건)

HIGH

F-01 · LLM 응답을 innerHTML로 직접 렌더링 (XSS 위험)

parseMarkdown의 결과물이 innerHTML에 주입되어, 서버 측 응답 조작 시 XSS로 이어질 수 있음
취약 위치
chatbot.js:668
취약 코드
// 스트리밍 토큰마다 innerHTML을 갱신
assistantText += event.content;
assistantBubble.innerHTML = parseMarkdown(assistantText);
공격 시나리오
API 서버가 침해되거나 중간자 공격(MITM)이 발생한 경우, SSE 스트림에 악의적인 HTML을 삽입할 수 있습니다. escapeHtml이 호출되지만 싱글 쿼트(')가 이스케이프되지 않으며, 마크다운 파서의 URL 정규식 [^)]+?이 인코딩된 페이로드를 통과시킬 수 있습니다. 예: ![x](https://x.com/x" onerror="alert(1)) 형태의 페이로드.
⚠ 현재 코드 문제
• escapeHtml이 ' 미이스케이프
• img/a 태그 생성 시 URL을 <속성값>에 직접 삽입
• innerHTML 재렌더링 시 이전 sanitize 결과 보장 없음
✅ 권고 조치
• escapeHtml에 ' → ' 추가
• DOMPurify 적용 또는 DOM API 기반 렌더러로 교체
• img src/alt를 setAttribute로 분리 설정
수정 예시
// 1) escapeHtml에 싱글 쿼트 추가
function escapeHtml(text) {
  return text
    .replace(/&/g, "&amp;")
    .replace(/</g,  "&lt;")
    .replace(/>/g,  "&gt;")
    .replace(/"/g,   "&quot;")
    .replace(/'/g,   "&#39;");  // 추가
}

// 2) 이미지는 DOM API로 직접 생성 (innerHTML 우회)
function createSafeImg(src, alt) {
  const img = document.createElement("img");
  img.setAttribute("src", src);   // 속성별 설정
  img.setAttribute("alt", alt);
  img.style.maxWidth = "100%";
  return img.outerHTML;
}
MEDIUM

F-02 · button_icon_url — 임의 외부 도메인 이미지 로드 허용

서버가 반환하는 아이콘 URL에 대한 도메인 검증이 없어 SSRF 프록시 또는 트래킹 픽셀에 악용 가능
취약 위치
chatbot.js:388–405
취약 코드
if (config.button_icon_url) {
  const iconUrl = config.button_icon_url.startsWith("http")
    ? config.button_icon_url          // ← 모든 외부 URL 허용
    : (() => { ... })();
  img.src = iconUrl;                  // 프로토콜·도메인 검증 없음
}
위험 시나리오
악의적인 서버가 button_icon_url을 공격자 서버로 설정하면, 위젯이 임베드된 모든 페이지에서 공격자 서버로 GET 요청이 발생합니다(IP 수집, 트래킹). 또한 내부망 URL을 지정하면 브라우저를 SSRF 프록시로 활용할 수 있습니다.
권고 수정
function isSafeIconUrl(url, apiUrl) {
  try {
    const u = new URL(url);
    const apiOrigin = new URL(apiUrl).origin;
    // 같은 origin 또는 https 한정 허용 (필요 시 allowlist 추가)
    return u.protocol === "https:" && u.origin === apiOrigin;
  } catch { return false; }
}

if (config.button_icon_url && isSafeIconUrl(config.button_icon_url, API_URL)) {
  img.src = config.button_icon_url;
}

F-03 · 입력 길이 제한 미적용 (DoS / 과도한 API 비용)

입력 필드에 maxlength 없음 — 무제한 길이의 메시지를 서버로 전송 가능
취약 위치
chatbot.js:342 (HTML) · chatbot.js:594 (sendMessage)
현재 코드
<!-- maxlength 속성 없음 -->
<input type="text" id="msg-input" placeholder="메시지를 입력하세요..." autocomplete="off" />

// sendMessage — 길이 검사 없음
const text = msgInput.value.trim();
if (!text || isStreaming) return;
⚠ 위험
수만 자 이상의 텍스트 전송 시
• LLM API 토큰 비용 급증
• 서버 처리 부하 (DoS 유발 가능)
✅ 권고
• input에 maxlength="2000" 추가
• sendMessage에서도 길이 확인
• 초과 시 UI 오류 메시지 표시
수정 예시
<!-- HTML -->
<input type="text" id="msg-input" maxlength="2000" ... />

// sendMessage
const MAX_MSG_LEN = 2000;
const text = msgInput.value.trim();
if (!text || isStreaming) return;
if (text.length > MAX_MSG_LEN) {
  showError(`메시지는 ${MAX_MSG_LEN}자 이하로 입력해 주세요.`);
  return;
}

F-04 · API_URL 검증 없음 — 스크립트 설정을 통한 데이터 탈취 가능

window.RagChatConfig.apiUrl을 임의 URL로 설정하면 사용자 메시지가 외부 서버로 전송됨
취약 위치
chatbot.js:17
취약 코드
const API_URL = cfg.apiUrl || "/rag/api/v1/chat";
// 프로토콜·호스트 검증 없음 — 공격자가 페이지를 XSS로 제어하거나
// 파트너사 임베드 환경에서 잘못된 URL을 주입할 수 있음
권고 수정
function sanitizeApiUrl(url) {
  if (!url) return "/rag/api/v1/chat";
  try {
    const u = new URL(url, location.href);
    if (u.protocol !== "https:" && u.protocol !== "http:")
      throw new Error("invalid protocol");
    return u.href;
  } catch {
    console.warn("[RagChat] 잘못된 apiUrl — 기본값 사용");
    return "/rag/api/v1/chat";
  }
}
const API_URL = sanitizeApiUrl(cfg.apiUrl);

F-05 · 클라이언트 측 Rate Limiting 미적용

isStreaming 플래그만으로는 자동화 도구에 의한 고속 메시지 전송을 막을 수 없음
설명
현재 코드는 스트리밍 중 중복 전송을 막는 isStreaming 플래그만 존재합니다. 이전 응답이 완료되면 즉시 다음 요청을 보낼 수 있어, 자동화 스크립트가 위젯을 통해 API 서버에 과부하를 줄 수 있습니다. 서버 측 rate limiting이 있어야 하지만, 위젯도 최소 클라이언트 쿨다운을 구현하는 것이 좋습니다.
권고 수정
let lastSentAt = 0;
const MIN_SEND_INTERVAL_MS = 500;  // 0.5초 쿨다운

async function sendMessage() {
  const now = Date.now();
  if (now - lastSentAt < MIN_SEND_INTERVAL_MS) return;
  lastSentAt = now;
  // ... 기존 로직
}

F-06 · session_id 미검증 — SSE 응답 조작 가능성

서버로부터 받은 session_id를 형식 검증 없이 그대로 저장하고 이후 요청에 사용
취약 위치
chatbot.js:651
현재 코드
if (event.type === "session") {
  sessionId = event.session_id;  // 형식 검증 없이 그대로 저장
}
권고 수정
const SESSION_ID_RE = /^[a-zA-Z0-9_-]{8,128}$/;

if (event.type === "session") {
  const sid = event.session_id;
  if (typeof sid === "string" && SESSION_ID_RE.test(sid)) {
    sessionId = sid;
  } else {
    console.warn("[RagChat] 비정상 session_id 수신 — 무시");
  }
}

F-07 · document 레벨 이벤트 리스너 — 호스트 페이지 간섭 가능성

Shadow DOM 밖의 document에 keydown/click 리스너를 등록해 호스트 이벤트 흐름에 영향을 줄 수 있음
취약 위치
chatbot.js:708–715
현재 코드
// Shadow DOM 밖 document에 직접 등록
document.addEventListener("click", (e) => {
  if (isOpen && !host.contains(e.target)) closeWidget();
});
document.addEventListener("keydown", (e) => {
  if (e.key === "Escape" && isOpen) closeWidget();
});
설명 및 권고
Escape 키 리스너는 호스트 페이지의 모달이나 다이얼로그 닫기 동작과 충돌할 수 있습니다. e.stopPropagation() 없이 등록하면 호스트의 핸들러도 실행됩니다. 위젯이 열린 상태에서만 동작하므로 실제 공격 위험도는 낮지만, { passive: true } 옵션 적용 및 컴포넌트 해제 시 removeEventListener 처리를 권고합니다.

F-08 · escapeHtml — 싱글 쿼트 미이스케이프

현재 구조에서는 직접 취약점이 아니지만, 추후 코드 변경 시 attribute injection 가능성 존재
취약 위치
chatbot.js:459–465
현재 코드
function escapeHtml(text) {
  return text
    .replace(/&/g, "&amp;")
    .replace(/</g, "&lt;")
    .replace(/>/g, "&gt;")
    .replace(/"/g, "&quot;");
    // ' 미이스케이프
}
권고
.replace(/'/g, "'") 한 줄을 추가하여 방어 깊이를 높입니다. 현재 parseMarkdown의 속성값은 모두 이중 쿼트를 사용하므로 즉각적인 위험은 없습니다.
INFO

F-09 · API Key가 HTTP 요청 헤더에 평문 전송

X-API-Key 헤더가 브라우저 개발도구에서 누구나 확인 가능 — 클라이언트 위젯의 구조적 한계
설명
위젯 특성상 API Key는 브라우저에 노출될 수밖에 없습니다. 이를 완화하려면 서버 측에서 ① API Key를 테넌트 식별에만 사용하고 ② 도메인 화이트리스트 검증을 강제하며 ③ Key를 주기적으로 로테이션하는 정책이 필요합니다. 현재 domain_validation.py가 존재하므로 이 경로를 유지·강화하는 것을 권고합니다.
권고
서버 측 도메인 검증(Referer / Origin 헤더 검사)을 반드시 활성화하고, 위젯 Key와 관리자 Key를 분리하여 권한 범위를 최소화합니다.
