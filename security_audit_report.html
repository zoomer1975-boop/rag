🔴 CRITICAL

Critical 수정 완료 C-01 · 채팅 메시지 길이 제한 없음 — DoS / 프롬프트 인젝션 ▼
파일apps/api/app/routers/chat.py
취약점 유형DoS, Prompt Injection
영향 범위모든 테넌트 채팅 엔드포인트

ChatRequest.message 필드에 길이 제한이 없어 악의적 사용자가 수십 MB 크기의 메시지를 전송해 LLM 토큰을 고갈시키거나, 매우 긴 지시문을 삽입해 시스템 프롬프트를 덮어쓰는 프롬프트 인젝션을 시도할 수 있었습니다. 또한 session_id가 임의의 문자열을 허용해 UUID 형식이 아닌 값으로 세션 조작 시도가 가능했습니다.

수정 전

class ChatRequest(BaseModel):
    message: str
    session_id: str | None = None

수정 후

class ChatRequest(BaseModel):
    message: str = Field(..., min_length=1, max_length=4000)
    session_id: str | None = Field(
        None,
        pattern=r"^[0-9a-f]{8}-[0-9a-f]{4}-[0-9a-f]{4}-[0-9a-f]{4}-[0-9a-f]{12}$",
    )
Critical 수정 완료 C-02 · 관리자 토큰 비교 시 타이밍 어택 취약점 ▼
파일apps/api/app/middleware/auth.py
취약점 유형Timing Side-Channel Attack
영향 범위관리자 API 전체 (테넌트 생성/삭제, 설정 변경 등)

관리자 토큰 검증에 Python의 일반 문자열 비교(!=)를 사용해 타이밍 어택이 가능했습니다. 공격자는 응답 시간 차이를 측정해 토큰을 한 문자씩 추측할 수 있습니다. secrets.compare_digest()는 일정 시간(constant-time) 비교를 수행해 이를 방지합니다.

수정 전

if not x_admin_token or x_admin_token != settings.admin_api_token:

수정 후

import secrets
...
if not x_admin_token or not secrets.compare_digest(x_admin_token, settings.admin_api_token):

🟠 HIGH

High 수정 완료 H-01 · 속도 제한(Rate Limiting) 설정만 존재하고 실제 적용 안 됨 ▼
파일apps/api/app/config.py, apps/api/app/main.py
취약점 유형Denial of Service, Brute Force
영향 범위채팅(/api/v1/chat), 인제스트(/api/v1/ingest) 엔드포인트

settings.rate_limit_requests와 rate_limit_window 값이 설정 파일에 정의되어 있었으나 어떤 미들웨어나 엔드포인트에도 적용되지 않았습니다. 이로 인해 LLM API 호출 비용을 고갈시키는 DoS 공격에 무방비 상태였습니다.

수정

# apps/api/app/middleware/rate_limit.py (신규 생성)
class RateLimitMiddleware(BaseHTTPMiddleware):
    """X-API-Key 기준 Redis 슬라이딩 윈도우 속도 제한.
    /api/v1/chat, /api/v1/ingest 경로에만 적용.
    Redis 장애 시 fail-open."""

# apps/api/app/main.py
app.add_middleware(
    RateLimitMiddleware,
    redis_url=settings.redis_url,
    limit=settings.rate_limit_requests,  # 기본: 60
    window=settings.rate_limit_window,   # 기본: 60초
)
High 수정 완료 H-02 · 아이콘 업로드 시 Content-Type 신뢰 — 파일 타입 우회 가능 ▼
파일apps/api/app/routers/tenants.py
취약점 유형Unrestricted File Upload
영향 범위위젯 아이콘 업로드 엔드포인트

클라이언트가 제공하는 Content-Type 헤더를 신뢰해 이미지 유형을 검증했습니다. 공격자는 PHP/JS/SVG 스크립트 파일을 image/png로 선언해 업로드할 수 있었습니다. Magic bytes(파일 시그니처) 검증으로 대체했습니다.

수정 전

if file.content_type not in ALLOWED_IMAGE_TYPES:
    raise HTTPException(...)

수정 후

def _detect_image_ext(data: bytes) -> str | None:
    if data[:8] == b"\x89PNG\r\n\x1a\n": return "png"
    if data[:3] == b"\xff\xd8\xff":      return "jpg"
    if data[:6] in (b"GIF87a", b"GIF89a"): return "gif"
    if data[:4] == b"RIFF" and data[8:12] == b"WEBP": return "webp"
    return None

# 업로드 핸들러
contents = await file.read()
ext = _detect_image_ext(contents)
if ext is None:
    raise HTTPException(status_code=400, ...)

🟡 MEDIUM

Medium 설계 트레이드오프 M-01 · CORS allow_origins=["*"] — 와일드카드 CORS ▼
파일apps/api/app/main.py
취약점 유형CORS Misconfiguration
영향 범위모든 API 엔드포인트

임베드 위젯 특성상 고객 사이트 어디서나 로드될 수 있어 와일드카드 CORS가 의도된 설계일 수 있습니다. 그러나 allow_credentials=False가 유지되어야 하고, 각 테넌트별 도메인 화이트리스트 검증이 is_origin_allowed()를 통해 채팅 엔드포인트에서 별도 수행되고 있습니다. 관리자 API(/api/v1/admin)는 별도의 X-Admin-Token 인증을 사용합니다.

권장 사항: CORS 자체보다 테넌트별 도메인 화이트리스트(현재 구현됨)가 더 실질적인 보호입니다. 와일드카드 유지는 허용되나 allow_credentials가 False로 유지되는지 지속적으로 확인하세요.

Medium 기존 경고 유지 M-02 · ADMIN_API_TOKEN 미설정 시 경고만 로깅 — 서비스 미차단 ▼
파일apps/api/app/config.py, apps/api/app/middleware/auth.py
취약점 유형Missing Authentication
영향 범위관리자 API 전체

ADMIN_API_TOKEN=""일 때 시작 시 경고만 로깅하고 서비스가 계속 실행됩니다. verify_admin()에서 토큰 미설정 시 503을 반환하는 로직이 이미 있어 실제 관리자 API 접근은 차단되지만, 프로덕션 배포 체크리스트에 명시해야 합니다.

권장 사항: 프로덕션 환경에서는 토큰 미설정 시 애플리케이션 시작을 거부하는 옵션을 고려하세요. 현재는 verify_admin()이 503을 반환하므로 관리자 API는 보호됩니다.

Medium 인지 필요 M-03 · 도구 실행기 로그에 URL/도구명 직접 출력 — 로그 인젝션 ▼
파일apps/api/app/services/tool_executor.py
취약점 유형Log Injection
영향 범위서버 로그 파일

외부 API 도구의 이름과 URL이 정제 없이 로그에 기록됩니다. 공격자가 줄바꿈 문자(\n)를 포함한 도구명을 생성하면 로그 파싱 도구를 오염시킬 수 있습니다.

권장 사항: 로그 출력 전 \n, \r 문자를 제거하거나 구조화 로깅(JSON 포맷)을 사용하세요. Python의 logging은 기본적으로 이를 처리하지 않습니다.

🟢 LOW

Low 경고 존재 L-01 · 불안전한 기본값 — SECRET_KEY, ADMIN_PASSWORD ▼
파일apps/api/app/config.py
취약점 유형Insecure Default Configuration

SECRET_KEY="change-me-to-a-random-secret-key", ADMIN_PASSWORD="change-me"가 기본값으로 설정되어 있습니다. 현재 시작 시 경고를 로깅하지만 서비스를 차단하지는 않습니다.

권장 사항: CI/CD 파이프라인에서 기본값 사용 여부를 자동 검사하고, .env.example에 강력한 값 생성 방법을 안내하세요. (openssl rand -hex 32)

Low 정보 제공 L-02 · LLM 프롬프트 인젝션 — 사용자 입력이 시스템 프롬프트와 분리 없이 결합 ▼
파일apps/api/app/services/rag.py
취약점 유형Prompt Injection (LLM specific)

사용자 쿼리가 RAG 컨텍스트와 함께 LLM에 전달될 때 사용자가 "이전 지시를 무시하고..." 같은 지시를 삽입할 수 있습니다. C-01 수정으로 메시지 길이가 4000자로 제한되어 공격 표면이 줄었으나, 완전한 방어는 LLM 레벨에서 구현해야 합니다.

권장 사항: 시스템 프롬프트와 사용자 입력 역할을 명확히 구분하는 프롬프트 가드레일(OpenAI의 role 분리 패턴 등)을 적용하세요.

✅ 안전 확인 항목

  • SQL Injection 방어: SQLAlchemy ORM 파라미터 바인딩을 사용하며 raw SQL 문자열 연결 없음 — 안전
  • XSS 방어 (위젯): chatbot.js에서 사용자 메시지는 textContent로, 어시스턴트 메시지는 escapeHtml() 후 innerHTML로 렌더링 — 안전
  • SSRF 방어: ssrf_guard.py에서 RFC1918/loopback/link-local/IPv6 사설 대역 차단, DNS 해석 후 검증, follow_redirects=False — 안전
  • 파일 업로드 (문서): UUID로 파일명 재생성, 확장자 화이트리스트, 크기 제한 — 안전
  • API Key 인증: SQLAlchemy ORM 파라미터 쿼리, 비활성 테넌트 차단 — 안전
  • 도메인 화이트리스트: Origin 헤더 파싱 후 scheme·hostname 검증, 빈 리스트 시 모든 도메인 허용(개발 모드) — 구현 견고
  • 경로 순회 방어: 정적 파일 서빙 시 /, \, . prefix 차단 — 안전
  • API 응답 크기 제한: Tool Executor에서 MAX_RESPONSE_BYTES = 100KB 제한 — 안전
  • 링크/이미지 XSS: 위젯 마크다운 파서가 이미지/링크 URL을 https?://로 제한, rel="noopener noreferrer" — 안전
  • 도메인 목록 제한: 테넌트당 최대 50개 도메인 상한(MAX_DOMAINS_PER_TENANT=50) — 안전
